## What the Issue Asks For

We want to:

1. **Export** a p5.js sketch as an SVG — true vector output, not raster pixels.
2. **Import** an SVG into p5.js and edit/manipulate it programmatically.

### Why It Matters

- **Pen plotters / digital fabrication** — Machines like Axidraw need vector paths (SVG), not pixels.
- **Infinite scalability** — SVG stays sharp at any zoom level.
- **Editability** — SVG files can be opened in Illustrator/Inkscape and individual shapes can be manipulated.
- **Processing parity** — Processing (Java) has excellent SVG export via `PShape` and `beginRecord()`/`endRecord()`. p5.js should match that.

### Current Workarounds

- [p5.js-svg](https://github.com/zenozeng/p5.js-svg) by `@zenozeng` — Third-party SVG runtime. Works but has performance issues and doesn't cover all use cases.
- [p5.plotSvg](https://github.com/golanlevin/p5.plotSvg) by `@golanlevin` — SVG exporter specifically for pen-plotting. Maintains normal canvas display.

## What SVG Actually Is

SVG (Scalable Vector Graphics) is an XML-based text format that describes images using **geometry and math**, not pixels.

A PNG stores millions of colored dots. An SVG stores instructions:

```xml
<svg width="400" height="400" xmlns="http://www.w3.org/2000/svg">
  <rect x="10" y="10" width="50" height="50" fill="red" stroke="black"/>
  <circle cx="200" cy="200" r="20" fill="blue"/>
  <line x1="0" y1="0" x2="100" y2="100" stroke="green" stroke-width="2"/>
</svg>
```

The p5 2D API maps almost 1:1 to SVG elements:

| p5 command                    | SVG element                                    |
| ----------------------------- | ---------------------------------------------- |
| `rect(10, 10, 50, 50)`        | `<rect x="10" y="10" width="50" height="50"/>` |
| `circle(200, 200, 40)`        | `<circle cx="200" cy="200" r="20"/>`           |
| `ellipse(100, 100, 60, 40)`   | `<ellipse cx="100" cy="100" rx="30" ry="20"/>` |
| `line(0, 0, 100, 100)`        | `<line x1="0" y1="0" x2="100" y2="100"/>`      |
| `triangle(0,0, 50,50, 100,0)` | `<polygon points="0,0 50,50 100,0"/>`          |
| `bezier(...)`                 | `<path d="M... C..."/>`                        |
| `fill(255, 0, 0)`             | `fill="rgb(255,0,0)"` attribute                |
| `stroke(0)`                   | `stroke="rgb(0,0,0)"` attribute                |
| `strokeWeight(3)`             | `stroke-width="3"` attribute                   |
| `translate(50, 50)`           | `<g transform="translate(50,50)">...</g>`      |
| `rotate(PI/4)`                | `<g transform="rotate(45)">...</g>`            |
| `push()/pop()`                | Nested `<g>` groups                            |

### Why You Can't Convert Canvas → SVG After the Fact

Canvas is a pixel grid. Once `rect()` paints pixels, the fact that it was a rectangle is **lost**. You can't reverse-engineer vector shapes from a grid of colors. That's why draw commands must be intercepted **before** (or as) they hit the canvas.

## Our Proposed Approach

### Phase 1: Snapshot SVG Export (start here)

**Core idea:** The sketch runs normally on canvas. When the user calls `saveSVG()`, we capture one frame's draw commands and build SVG from them.

**Flow:**

```
User's sketch runs normally on canvas (P2D)
         │
         ├──→ Canvas shows pixels (visual preview, like normal)
         │
         └──→ When saveSVG() is called, recording is enabled for one frame
              │
              ├──→ Each draw call (rect, circle, line, etc.) does its
              │    normal canvas work AND builds SVG nodes
              │
              └──→ At end of frame, serialize SVG and download .svg file
                   Recording turns off — zero overhead returns
```

**User experience:**

```js
function setup() {
  createCanvas(400, 400); // normal canvas, nothing changes
}

function draw() {
  background(255);
  fill(255, 0, 0);
  rect(10, 10, 50, 50);
  fill(0, 0, 255);
  circle(200, 200, 40);

  if (frameCount === 1) {
    saveSVG("myDrawing"); // exports this frame as SVG
  }
}
```

**Key design decisions:**

- **Zero overhead normally** — SVG is only built during the export frame, not every frame.
- **Snapshot, not continuous** — Only captures one frame when asked. No accumulation problem.
- **Canvas stays primary** — No change to how sketches display. SVG is a side-output.

### Phase 2 (future): Full SVG Renderer

`createCanvas(400, 400, SVG)` — The `<svg>` element IS the display. Every draw call creates SVG nodes directly. No canvas involved. Bigger lift, comes later.

### The Visitor Pattern (Extension Point)

In dev-2.0, shapes use a **visitor pattern**:

- `Shape` — collection of contours and primitives
- `PrimitiveVisitor` — abstract class with `visitAnchor()`, `visitLineSegment()`, `visitBezierSegment()`, `visitPoint()`, `visitLine()`, `visitTriangle()`, `visitQuad()`, etc.
- `PrimitiveToPath2DConverter` — concrete visitor that builds browser `Path2D` objects from shape primitives

The `Renderer2D.drawShape()` method:

```js
drawShape(shape) {
    const visitor = new PrimitiveToPath2DConverter({strokeWeight});
    shape.accept(visitor);  // walks all primitives, builds Path2D
    ctx.fill(visitor.path);
    ctx.stroke(visitor.path);
}
```

### What We Need to Build

#### 1. `PrimitiveToSVGConverter`

New visitor in `src/shape/custom_shapes.js`. Extends `PrimitiveVisitor`. Each `visit*` method creates a plain typed object instead of `Path2D` calls.

**Constructor takes all style info upfront** (unlike Path2D converter which only takes strokeWeight):

```js
constructor({ strokeWeight, strokeColor, fillColor, strokeCap, strokeJoin });
```

**Output stored as plain typed objects** in `this.elements = []`:

```js
{ type: "line", x1: 0, y1: 0, x2: 100, y2: 100, stroke: "rgb(0,0,0)", strokeWidth: 2 }
{ type: "rect", x: 10, y: 10, width: 50, height: 50, fill: "rgb(255,0,0)", stroke: "none" }
{ type: "circle", cx: 200, cy: 200, r: 20, fill: "rgb(0,0,255)" }
{ type: "polygon", points: "0,0 50,50 100,0", fill: "rgb(0,255,0)" }
{ type: "path", d: "M 0 0 L 100 100 C ...", fill: "none", stroke: "rgb(0,0,0)" }
{ type: "g", transform: "translate(50,50)", children: [...] }
```

No Zod validation — these objects are created by our own code, not user input. Plain objects keep it simple and fast.

**How visit methods work — examples:**

`visitLine`:

- Extracts `x0, y0, x1, y1` from `line.vertices` (same as Path2D converter)
- Pushes `{ type: "line", x1: x0, y1: y0, x2: x1, y2: y1, stroke, strokeWidth }` to `this.elements`

`visitQuad`:

- Extracts 4 vertices (same as Path2D converter)
- No `<quad>` in SVG, so maps to `<polygon>`
- Pushes `{ type: "polygon", points: "x0,y0 x1,y1 x2,y2 x3,y3", fill, stroke }`

Path primitives (`visitAnchor`, `visitLineSegment`, `visitBezierSegment`):

- Build a `d` attribute string: `M x y` (move), `L x y` (line), `C x1 y1 x2 y2 x3 y3` (cubic bezier), `Q x1 y1 x2 y2` (quadratic), `Z` (close)
- Combine into one `{ type: "path", d: "M... L... C...", fill, stroke }`

#### 2. SVG Recording in Renderer2D

When export flag is set, each draw method (`rect()`, `ellipse()`, `line()`, `arc()`, `drawShape()`, etc.) also appends SVG typed objects to an in-memory array.

#### 3. `saveSVG()` Function

New export function (in `src/image/image.js` or `src/io/files.js`):

- Sets the recording flag
- After the frame completes, a serializer converts the typed objects array → SVG XML string
- Downloads the `.svg` file using the existing `downloadFile()` utility
- Clears the recording flag and elements array

#### 4. Serializer

Takes `elements[]` array, produces SVG XML string:

```xml
<svg width="400" height="400" xmlns="http://www.w3.org/2000/svg">
  <!-- serialized from elements array -->
  <rect x="10" y="10" width="50" height="50" fill="rgb(255,0,0)"/>
  <circle cx="200" cy="200" r="20" fill="rgb(0,0,255)"/>
</svg>
```

Switches on `element.type` to produce the right XML tag and attributes.

### What We Do NOT Need to Change

- `createCanvas()` — No new renderer constant needed for Phase 1
- `p5.Renderer.js` — Base class stays the same
- Existing sketch behavior — Everything works as before unless `saveSVG()` is called
